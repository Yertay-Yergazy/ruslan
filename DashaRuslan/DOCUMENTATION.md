# Определения

В настоящем дипломном проекте применяются следующие термины с соответствующими определениями:

**Веб-приложение** — программное обеспечение, доступное через браузер по протоколу HTTP/HTTPS без установки на стороне клиента.

**ORM (Object-Relational Mapping)** — технология, обеспечивающая работу с реляционной базой данных посредством объектов языка программирования.

**Blueprint** — модульная единица Flask-приложения, объединяющая маршруты, формы и шаблоны одного функционального блока.

**CSRF (Cross-Site Request Forgery)** — вид атаки, при которой вредоносный ресурс инициирует запросы от имени аутентифицированного пользователя.

**Миграция базы данных** — управляемое версионирование схемы базы данных с возможностью наката и отката изменений.

**Сессия** — механизм сохранения состояния пользователя между HTTP-запросами.

**Слот** — минимальная единица времени (30 минут) для записи на услугу в расписании мастера.

**Роль пользователя** — набор прав доступа, определяющий допустимые действия пользователя в системе.

# Введение

Автоматизация процессов обслуживания клиентов является одним из ключевых направлений цифровизации малого и среднего бизнеса. Предприятия сферы автосервиса, работающие без специализированных систем управления, сталкиваются с рядом типичных проблем: ручное ведение записей приводит к ошибкам и двойным бронированиям, отсутствие истории обслуживания затрудняет работу с постоянными клиентами, а непрозрачность загрузки мастеров снижает эффективность использования ресурсов.

Целью настоящего дипломного проекта является разработка веб-приложения для управления автосервисом, автоматизирующего процессы записи клиентов, управления персоналом и каталогом услуг.

Для достижения поставленной цели решаются следующие задачи:

- анализ предметной области и формирование требований к системе;
- выбор и обоснование технологического стека;
- проектирование архитектуры и схемы базы данных;
- реализация клиентского и административного модулей;
- обеспечение безопасности приложения.

Практическая значимость работы состоит в том, что разработанная система может быть применена в реальных условиях малого автосервиса без значительных затрат на инфраструктуру.

# 1. Описание предметной области

## 1.1 Постановка задачи

Объектом автоматизации является малый автосервис с персоналом от 3 до 15 мастеров, обслуживающий частных клиентов. Типичная проблематика таких предприятий включает:

- запись клиентов осуществляется по телефону или в мессенджерах без единой системы;
- расписание мастеров ведётся в бумажных журналах или в таблицах;
- отсутствует возможность онлайн-записи для клиентов;
- история обслуживания автомобиля не сохраняется в структурированном виде.

Разрабатываемая система должна устранить перечисленные недостатки и предоставить следующий функционал:

- публичный сайт-витрину с каталогом услуг и возможностью онлайн-записи;
- личный кабинет клиента с историей записей;
- административную панель для управления записями, услугами и персоналом;
- автоматическую проверку доступности мастера при бронировании.

## 2.1 Серверная часть

- декларативное описание схемы непосредственно в коде;

Система миграций обеспечивает версионирование схемы базы данных. Каждое изменение схемы фиксируется в виде скрипта миграции с поддержкой наката и отката.

**Flask-Login 0.6.3**

Библиотека управляет пользовательскими сессиями. Декоратор `@login_required` обеспечивает защиту маршрутов. Объект `current_user` доступен во всех частях приложения и шаблонах.

**Flask-WTF 1.2.1**

Обеспечивает серверную валидацию форм и автоматическую генерацию CSRF-токенов. Декларативное описание форм позволяет повторно использовать их для операций создания и редактирования.

**SQLite**

Для хранения данных применяется встроенная файловая СУБД SQLite. Обоснование выбора:

- не требует отдельного сервера базы данных;
- вся БД хранится в одном файле, что упрощает развёртывание и резервное копирование;
- производительности достаточно для нагрузки малого автосервиса.

При необходимости масштабирования SQLite заменяется на PostgreSQL изменением параметра `SQLALCHEMY_DATABASE_URI` в конфигурационном файле.

**Gunicorn 22.0.0**

Встроенный сервер Flask предназначен исключительно для разработки. Для промышленного развёртывания применяется Gunicorn — WSGI-сервер с поддержкой нескольких рабочих процессов, совместимый с обратным прокси-сервером Nginx.

# 3. Архитектура системы

## 3.1 Общая архитектура

Приложение реализовано по паттерну MVC (Model-View-Controller) в адаптации для Flask:

- **Model** — файл `app/models.py`, содержащий ORM-модели SQLAlchemy;
- **View** — каталог `app/templates/`, содержащий Jinja2-шаблоны;
- **Controller** — файлы `app/*/routes.py`, содержащие Blueprint-маршруты.

Такое разделение обеспечивает независимость слоёв: изменение логики отображения не затрагивает бизнес-логику, а изменение схемы БД локализовано в моделях.

## 3.2 Паттерн Application Factory

Инициализация приложения вынесена в функцию `create_app()` в файле `app/__init__.py`. Данный подход обеспечивает следующие преимущества:

- возможность создания нескольких экземпляров приложения с разными конфигурациями (разработка, тестирование, промышленная эксплуатация);
- устранение циклических импортов через отложенную инициализацию расширений;
- удобное автономное тестирование без запуска HTTP-сервера.

Порядок инициализации в `create_app()`:

1. Загрузка конфигурации из класса `Config`;
2. Инициализация расширений: `db`, `login_manager`, `migrate`, `csrf`;
3. Регистрация Blueprint-модулей;
4. Регистрация глобальных фильтров Jinja2 (`format_time`, `format_date`, `format_price`).

## 3.3 Blueprint-архитектура

Приложение разделено на четыре независимых Blueprint-модуля, каждый из которых изолирует маршруты, формы и логику своего функционального блока. Структура модулей представлена в таблице 5.

## 3.4 Алгоритм поиска свободных временных слотов

Алгоритм реализован в функции `get_busy_slots()` и `is_slot_available()` файла `app/utils.py`. При получении запроса к эндпоинту `/api/available-slots` выполняется следующая последовательность шагов:

1. Определить день недели для указанной даты и найти запись расписания (`WorkSchedule`) мастера. Если запись отсутствует или флаг `is_working = False` — вернуть пустой список.

2. Получить из таблицы `Booking` все активные записи мастера на указанную дату (статусы `pending`, `confirmed`, `in_progress`). Для каждой записи отметить занятыми все 30-минутные блоки в диапазоне `[start_time, end_time)`.

3. Выполнить итерацию по временному диапазону расписания с шагом 30 минут. Для каждого кандидата-слота проверить, что все блоки в диапазоне `[slot, slot + duration)` отсутствуют в множестве занятых.

4. Вернуть список свободных слотов в формате `HH:MM`.

Сложность алгоритма составляет O(n × k), где n — количество существующих записей мастера на дату, k — количество генерируемых слотов. При типичной нагрузке (до 20 записей в день, до 24 слотов) вычисление занимает менее 1 миллисекунды.

1. клиент выбирает услугу из каталога или непосредственно на форме записи;
2. система динамически загружает список мастеров, выполняющих данную услугу;
3. клиент выбирает мастера и дату;
4. система возвращает список свободных временных слотов через API;
5. клиент выбирает время, указывает информацию об автомобиле и подтверждает запись;
6. создаётся объект `Booking` со статусом `pending`.

При успешной аутентификации пользователи с ролью `admin` перенаправляются на административную панель, пользователи с ролью `client` — на главную страницу. Опция «Запомнить меня» обеспечивает сохранение сессии на 7 дней.

## 4.3 Административная панель

Доступ к административной панели (`/admin`) разрешён исключительно пользователям с ролью `admin`. Защита реализована декоратором `@admin_required`.

**Дашборд** отображает сводную статистику: общее число записей, записи за сегодня и за текущий месяц, число ожидающих подтверждения, выручку, число клиентов и сотрудников. Дополнительно выводится график выручки за последние 6 месяцев и распределение записей по статусам.

**Управление записями** обеспечивает просмотр всех бронирований с пагинацией (20 записей на страницу), фильтрацию по статусу и дате, полнотекстовый поиск по имени клиента, номеру записи и модели автомобиля, а также изменение статуса записи и добавление внутренних заметок.

**Управление услугами** реализует полный CRUD для категорий услуг (с поддержкой иконок Font Awesome и slug-идентификаторов) и для самих услуг (с загрузкой изображений, флагами активности и размещения на главной странице). При попытке удаления услуги, связанной с существующими записями, выполняется мягкое удаление — деактивация.

**Управление сотрудниками** включает создание сотрудника с автоматическим созданием учётной записи пользователя роли `employee`, привязку к услугам (отношение многие ко многим), управление расписанием работы по дням недели. По умолчанию создаётся расписание с понедельника по пятницу с 09:00 до 18:00. Удаление сотрудника, имеющего записи, заменяется деактивацией.

**Управление клиентами** предоставляет список клиентов с поиском, детальную карточку с историей записей и функцию блокировки учётной записи.

**Управление отзывами** позволяет публиковать или скрывать отзывы клиентов, а также удалять их при необходимости.

# 5. Проектирование базы данных

## 5.1 Структура таблиц

База данных включает семь таблиц, описанных в таблице 9.

Таблица 9 — Таблицы базы данных

| Таблица | Назначение | Основные связи |
|:--------|:-----------|:---------------|
| `users` | Все пользователи системы | → `employees` (1:1), → `bookings` (1:N), → `reviews` (1:N) |
| `employees` | Профили сотрудников | → `users` (N:1), → `work_schedules` (1:N), → `bookings` (1:N) |
| `employee_services` | Связь мастер-услуга | M:N через `employees` и `services` |
| `service_categories` | Категории услуг | → `services` (1:N) |
| `services` | Услуги автосервиса | → `bookings` (1:N), ← `service_categories` (N:1) |
| `work_schedules` | Рабочее расписание мастеров | → `employees` (N:1) |
| `bookings` | Записи клиентов | → `users` (N:1), → `employees` (N:1), → `services` (N:1) |
| `reviews` | Отзывы клиентов | → `bookings` (1:1), → `users` (N:1) |

Модель `User` является центральной для системы аутентификации. Она унаследована от `UserMixin` библиотеки Flask-Login, что обеспечивает интеграцию с механизмом сессий. Роль пользователя (`admin`, `employee`, `client`) определяет доступные действия в системе.

Модель `Employee` является расширением `User` для сотрудников: один пользователь имеет не более одного профиля сотрудника (отношение один к одному). Профиль хранит специализацию, биографию и стаж. Рейтинг (`avg_rating`) и количество отзывов (`reviews_count`) вычисляются динамически по связанным записям.

Модель `Booking` является ключевой бизнес-сущностью. Она хранит уникальный номер записи формата `AS{YYYYMMDD}{6 символов UUID}`, связывает клиента, мастера и услугу, фиксирует дату, время начала и окончания, итоговую стоимость и информацию об автомобиле.

# 6. Обеспечение безопасности

Разработанная система реализует комплекс мер по обеспечению безопасности, охватывающих основные угрозы согласно классификации OWASP Top 10. Описание угроз и применяемых мер представлено в таблице 11.

Таблица 11 — Меры обеспечения безопасности

| Угроза | Применяемая мера защиты |
|:-------|:------------------------|
| SQL-инъекции | SQLAlchemy ORM применяет параметризованные запросы; прямые SQL-строки не используются |
| XSS (межсайтовый скриптинг) | Jinja2 автоматически экранирует весь вывод в шаблонах |
| CSRF | Flask-WTF генерирует и проверяет токены для каждой формы |
| Несанкционированный доступ к панели | Декоратор `@admin_required` возвращает HTTP 403 |
| Незащищённые пароли | Werkzeug PBKDF2-SHA256 с солью |
| Path Traversal при загрузке файлов | `secure_filename()` + белый список расширений + UUID-имена |
| Доступ к чужим данным | Проверка `booking.client_id != current_user.id` → HTTP 403 |
| Доступ без аутентификации | Декоратор `@login_required` + редирект на страницу входа |
| Брутфорс учётных записей | Блокировка аккаунта администратором (`is_active = False`) |

Хранение паролей реализовано через функцию `generate_password_hash()` библиотеки Werkzeug, использующей алгоритм PBKDF2-SHA256 с автоматически генерируемой солью. Проверка пароля при входе осуществляется функцией `check_password_hash()`, что полностью исключает хранение и сравнение паролей в открытом виде.

Загрузка файлов защищена тремя уровнями: функция `secure_filename()` устраняет опасные символы из имени файла; белый список допустимых расширений (`png`, `jpg`, `jpeg`, `gif`, `webp`) предотвращает загрузку исполняемых файлов; итоговое имя генерируется как UUID, что исключает предсказуемость пути к файлу.

# Приложение 1. Исходный код модуля utils.py

Модуль `app/utils.py` содержит вспомогательные функции и декораторы, используемые во всех Blueprint-модулях приложения.

```python
import os
import uuid
from datetime import datetime, time, timedelta
from functools import wraps
from flask import current_app, abort
from flask_login import current_user
from werkzeug.utils import secure_filename


def admin_required(f):
    @wraps(f)
    def decorated_function(*args, **kwargs):
        if not current_user.is_authenticated or not current_user.is_admin():
            abort(403)
        return f(*args, **kwargs)
    return decorated_function


def save_image(file, subfolder=''):
    if file and allowed_file(file.filename):
        ext = file.filename.rsplit('.', 1)[1].lower()
        filename = f"{uuid.uuid4().hex}.{ext}"
        upload_path = os.path.join(
            current_app.config['UPLOAD_FOLDER'], subfolder
        )
        os.makedirs(upload_path, exist_ok=True)
        file.save(os.path.join(upload_path, filename))
        return (os.path.join(subfolder, filename).replace('\\', '/')
                if subfolder else filename)
    return None


def generate_booking_number():
    now = datetime.utcnow()
    return f"AS{now.strftime('%Y%m%d')}{uuid.uuid4().hex[:6].upper()}"


def get_busy_slots(employee_id, date, service_duration,
                   exclude_booking_id=None):
    from app.models import Booking
    query = Booking.query.filter(
        Booking.employee_id == employee_id,
        Booking.booking_date == date,
        Booking.status.in_(['pending', 'confirmed', 'in_progress'])
    )
    if exclude_booking_id:
        query = query.filter(Booking.id != exclude_booking_id)
    busy = set()
    for booking in query.all():
        slot_start = datetime.combine(date, booking.start_time)
        slot_end = datetime.combine(date, booking.end_time)
        cursor = slot_start
        while cursor < slot_end:
            busy.add(cursor.time())
            cursor += timedelta(minutes=30)
    return busy


def is_slot_available(employee_id, date, start_time_obj, service_duration):
    from app.models import WorkSchedule
    schedule = WorkSchedule.query.filter_by(
        employee_id=employee_id,
        day_of_week=date.weekday(),
        is_working=True
    ).first()
    if not schedule:
        return False
    end_time_obj = (datetime.combine(date, start_time_obj) +
                    timedelta(minutes=service_duration)).time()
    if (start_time_obj < schedule.start_time
            or end_time_obj > schedule.end_time):
        return False
    busy = get_busy_slots(employee_id, date, service_duration)
    cursor = datetime.combine(date, start_time_obj)
    service_end = datetime.combine(date, end_time_obj)
    while cursor < service_end:
        if cursor.time() in busy:
            return False
        cursor += timedelta(minutes=30)
    return True
```
